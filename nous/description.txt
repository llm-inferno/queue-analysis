Background

queue-analysis is a Go REST server that solves a state-dependent Markovian queueing
model of a vLLM inference server (see pkg/queue/mm1modelstatedependent.go,
pkg/service/analyzer.go). Two HTTP endpoints on :8080:

  POST /solve   — analyze the queue at a given (RPS, MaxBatchSize, ...) and return
                  Throughput, AvgITL, AvgTTFT, AvgRespTime, MaxRPS, etc.
  POST /target  — given (MaxBatchSize, AvgInputTokens, AvgOutputTokens, alpha, beta,
                  gamma, MaxQueueSize, TargetTTFT, TargetITL), return the maximum RPS
                  that satisfies AvgITL <= TargetITL and AvgTTFT <= TargetTTFT. The
                  returned RPS is in the "throughput" field of the response.
                  Returns HTTP 400 when the (M, target-set) pair is infeasible.

Schemas: examples/problem-data.json (input), examples/solution-target.json (output).

NOTE: the analyzer's service-time formula was corrected in commit 99024df (it had
double-counted the focal request's work). Any prior NOUS findings for this system
predate that fix and are NOT load-bearing — derive afresh from the primitives below.

Analytic primitives

The per-iteration mean times are closed-form in the inputs. With
  n  = AvgInputTokens, m = AvgOutputTokens, alpha/beta/gamma the service constants,
  B  = batch size (= the MaxBatchSize variable M),
  nc = number of chunked-prefill iterations at B (a step function of B that depends on
       the fixed token budget MaxNumTokens = 8192; nc = 1 until B is large enough that
       prefill must be split into chunks):

  wPrefill = (beta + gamma*(nc+1)/2) * n
  wDecode  = beta*m + gamma*m*(n + (m+1)/2)
  delta    = (wPrefill + wDecode) / (nc + m)
  bg(B)    = max(0, alpha + (B-1)*delta)          # background iteration time

  itl(B)     = bg(B) + beta + gamma*(n + (m+1)/2)             # mean inter-token latency
  prefill(B) = nc*bg(B) + wPrefill                            # prefill component of TTFT
  tau(B)     = (nc + m) * (alpha + B*delta)                   # mean service time

These give per-iteration / per-request mean times. /target composes them through a
state-dependent M/M/1 solver (queue wait is added to prefill(B) to form TTFT, and the
solver finds the max RPS meeting both targets under the queue capacity Q = MaxQueueSize).

These formulas are available as an importable module: nous/harness/formulas.py exposes
num_iterations_per_prefill(B, params), itl(B, params), ttft_prefill(B, params),
tau(B, params), and delta(B, params), where params is the scenario dict described under
Harness contract. The module is parity-checked against the Go analyzer source. Import
and use it rather than re-deriving the arithmetic — but DO show your derivation of any
predictor in your iter-1 report so the analytic story is preserved.

Problem definition

For fixed inputs (alpha, beta, gamma, AvgInputTokens, AvgOutputTokens, MaxQueueSize,
TargetITL, TargetTTFT), define

    f(M) = throughput returned by /target when maxBatchSize = M.

Find  M* = argmax_{M in [M_min, M_max]} f(M).   Fixed: M_min = 1, M_max = 256.

A single /target call evaluates f at one M. Use the analytic primitives to derive a
predictor M-hat(params) where possible (solve each binding constraint — ITL, TTFT,
queue — in isolation), then refine via a small number of /target calls if needed.

Scenarios

Six named dev scenarios span the regimes. alpha/beta/gamma vary PER scenario (they are
swept on the realistic ratio manifold beta ~ alpha/240, gamma ~ alpha/24000). Canonical
JSON: nous/scenarios.json. Each scenario carries a `regime` label naming which
constraint binds at the peak.

  name        regime      alpha beta    gamma     AvgIn AvgOut ITL  TTFT  Q
  baseline    crossover   8.0   0.033   0.000333  256   1024   20   60    128
  itl-only    itl-only    8.0   0.033   0.000333  256   4096   15   400   128
  ttft-only   ttft-only   8.0   0.033   0.000333  1024  256    60   80    128
  unbounded   unbounded   4.0   0.017   0.000167  64    128    200  2000  4
  alpha-low   crossover   4.0   0.017   0.000167  256   1024   20   60    128
  alpha-high  crossover   16.0  0.067   0.000667  256   1024   20   60    128

Regime meanings: itl-only = ITL constraint binds across the range (f plateaus once ITL
saturates); ttft-only = TTFT binds; crossover = TTFT binds at small M and ITL at large M,
peak at the transition; unbounded = no constraint binds in [1,256], so M* = M_max.

A larger benchmark grid exists at nous/scenarios_benchmark.json (30 scenarios sampled
over the parameter ranges, regime label "bench"). It is NOT enumerated here and is for
the final iteration only — iters 1-4 work on the six dev scenarios above.

Harness contract

The harness lives at nous/harness/. Strategies (candidate algorithms) are Python files
under nous/harness/strategies/. Every strategy exposes:

    def search(target_eval, params: dict, m_min: int, m_max: int) -> int

  target_eval(m) posts to /target for that maxBatchSize, returns the parsed response
    dict, and counts the call. The strategy cannot bypass the wrapper.
  params is the scenario inputs as a dict:
    {alpha, beta, gamma, AvgInputTokens, AvgOutputTokens, targetITL, targetTTFT,
     maxQueueSize}
    — exactly the keys nous/harness/formulas.py consumes, so a strategy can pass params
    straight into the primitives to compute its predictor.

Prior strategies (predictor_*.py, adaptive_*.py, ratio_binary_search.py,
example_linear_scan.py) accept-and-ignore params; they are retained as a
parameter-blind BASELINE cohort for the final Pareto comparison. New formula-guided
work goes in new strategy files.

Infeasibility: if /target returns HTTP 400 for a particular M, the oracle returns
{"throughput": 0.0} and does NOT count it against the call budget. Treat 0-throughput
points as infeasible regions.

CLI:
    python -m nous.harness.run \
        --scenarios nous/scenarios.json \
        --strategy nous/harness/strategies/<name>.py \
        --out <results-path>.json

Outputs one record per scenario:
  M_chosen, calls, throughput_chosen, M_truth, throughput_truth,
  gap_throughput_rel, gap_M, wall_clock_seconds, internal_solve_calls.

The harness ALWAYS makes one extra confirmatory eval_(m_chosen) call after the strategy
returns, so calls = strategy_calls + 1. Strategy budgets must account for this +1.

Pareto axes and scoring

  calls               number of /target calls the strategy made (lower is better).
                      A 400-response (infeasibility) does NOT count toward this budget.
  gap_throughput_rel  (truth - chosen) / truth  (>= 0; lower is better; reported as
                                                 0 when truth = 0).
  gap_M               |M_chosen - M_truth|      (CONCURRENCY metric — see below: how far
                                                 the provisioned batch size sits above M*)

Aggregate across scenarios as both worst-case (max) and mean.

CONCURRENCY EFFICIENCY (primary objective from iter-4 onward). Empirically (validated
iter-3 on dev, iter-5 on the benchmark) f(M) is monotone-rising then FLAT: any M >= M*,
INCLUDING the trivial constant M = M_max = 256, achieves peak throughput within ~3% in a
single call. So gap_throughput_rel ALONE cannot reward a precise predictor — a dumb
M_max ties every strategy on throughput. The deployed value of M-hat is therefore
CONCURRENCY: the SMALLEST M that achieves near-peak throughput (the plateau-onset M*).
Over-provisioning to M_max wastes batch slots / KV-cache memory for zero throughput gain.

  - Target: onset M* = smallest M with gap_throughput_rel <= epsilon  (epsilon = 0.02).
  - Headline Pareto axes are now (calls, gap_M), subject to gap_f <= epsilon. gap_M is
    co-primary with calls; gap_f is a feasibility constraint to stay under, NOT the thing
    to win — landing anywhere on the plateau already scores gap_f ~ 0.
  - A strategy WINS iff, at gap_f <= epsilon, it Pareto-dominates BOTH baselines on
    (calls, gap_M): naive_max (constant M_max — 1 call but gap_M ~ M_max - M*) AND
    naive_ternary (blind search — ~25 calls). formula_guided must achieve this by finding
    the onset M*, NOT by seeding at M_max.

Scoring compares M_chosen against per-scenario truth caches (dev:
nous/cache/truth-<name>.json; benchmark: nous/cache/bench/<idx>.json), held by the
harness. The truth M*/f* are not pre-disclosed in this brief — discover f(M) via your
own /target calls and your derivation.

Stage plan

Five iterations. observe = reason + report, NO code changes. code_changes = add files
under nous/harness/strategies/. Stay within the current iteration's stage; produce the
listed output before advancing.

  Iter 1  observe       Derive M-hat(params) closed-form from the ITL/TTFT/tau
                        primitives. Solve each binding constraint in isolation.
                        Output: symbolic M_ITL, M_TTFT, M_queue and a rule to combine
                        them, with explicit assumptions. No oracle calls.
                        Gate: closed form for >= 1 binding case, with derivation; open
                        cases listed as "needs case analysis."
  Iter 2  observe       Map regime structure: where in params-space does each
                        constraint bind? Tests "are 4 groups enough?" directly.
                        Output: regime partition (<= 5 cells) with boundary equations
                        in params; per-cell M-hat. Pure analysis on the symbolic forms.
  Iter 3  observe       Validate M-hat against /target on the six dev scenarios.
                        Oracle calls allowed here (validation, NOT counted against any
                        algorithm budget). Output: per-scenario error table
                        gap_M = |M-hat - M_truth|, gap_f = (f_truth - f_Mhat)/f_truth.
                        Gate: worst-case gap_f <= 20% on >= 4/6 scenarios. Decide:
                        predictor alone, or predictor + search refinement.
  Iter 4  code_changes  Implement a CONCURRENCY-OPTIMAL formula-guided search that finds
                        the plateau ONSET M* (smallest M with gap_f <= epsilon=0.02), NOT
                        the bracket upper endpoint. Recommended mechanism: one probe at a
                        high seed to estimate f*, then BINARY-SEARCH DOWNWARD in
                        [M_ITL, seed] for the smallest M with f(M) >= (1-epsilon)*f*
                        (monotone predicate -> ~6-8 calls). Add formula_guided.py plus
                        baselines naive_ternary.py and naive_max.py (constant M_max).
                        Gate: on the dev set, worst-case calls <= 8 AND gap_f <= epsilon
                        AND formula_guided's worst-case gap_M strictly beats BOTH
                        naive_max and naive_ternary (i.e. it lands near M*, low gap_M).
                        NOTE: seeding high (gap_M large) is the prior failure mode — the
                        win is LOW gap_M, not just low gap_f.
  Iter 5  code_changes  Run on the benchmark grid (nous/scenarios_benchmark.json).
                        Pareto-compare on (calls, gap_M) subject to gap_f <= epsilon:
                        formula_guided vs naive_max vs naive_ternary vs the prior baseline
                        cohort. Per-regime breakdown (regime x worst calls, worst gap_M).
                        Headline claim to test: formula_guided Pareto-dominates BOTH naive
                        baselines on (calls, gap_M). Honest comparison report.

Out of scope

  - Changes to the Go analyzer code or the /solve / /target endpoints (only
    nous/harness/* and nous/harness/strategies/*).
  - Online / streaming variants of the algorithm.
  - Iter-4 strategies stay as single Python files under nous/harness/strategies/.
